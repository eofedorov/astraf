# ТЗ: расчёт total climb для велотрекера

## Цель

Добавить в приложение метрику **total climb / набор высоты** — суммарное количество метров подъёма за поездку.

---

## Что такое total climb

Total climb — это сумма всех значимых положительных изменений высоты за поездку.

Пример:

```text
100 м → 110 м → 105 м → 120 м
```

Набор высоты:

```text
+10 м + 15 м = 25 м
```

Спуски не вычитаются.

---

## Основная проблема

Высота из GPS шумная. Даже на ровной дороге altitude может прыгать на несколько метров.

Нельзя считать так:

```text
если новая высота выше предыдущей → добавить разницу
```

Это приведёт к завышенному total climb.

Нужен простой фильтр высоты.

---

# Функциональные требования

## 1. Хранение высоты

Для каждой GPS-точки нужно сохранять altitude, если Android Location его предоставляет.

Для точки хранить:

```text
latitude
longitude
timestamp
accuracy
altitude
hasAltitude
```

Если altitude отсутствует, точка всё равно может использоваться для трека, но не участвует в расчёте набора высоты.

---

## 2. Использовать только accepted GPS points

Total climb должен считаться только по точкам, которые уже прошли GPS-фильтрацию.

Нельзя учитывать:

- rejected GPS points;
- точки с плохой accuracy;
- точки между сегментами;
- точки во время паузы;
- точки до первого хорошего GPS fix.

---

## 3. Как считать подъём между сегментами

GPS gap не должен автоматически разрывать расчёт total climb.

Разрешается продолжать total climb, если:
abs(newAltitude - oldAltitude) <= MAX_REASONABLE_ALTITUDE_JUMP
MAX_REASONABLE_ALTITUDE_JUMP например 50

После восстановления GPS:
* если изменение altitude выглядит реалистичным — продолжать расчёт climb;
* если изменение altitude слишком большое — сбросить elevation continuity и начать новый baseline без добавления climb.

---

## 4. Сглаживание altitude

Перед расчётом total climb высоту нужно сглаживать.

Для первого этапа достаточно moving average по последним N accepted-точкам с altitude.

Пример конфига:

```kotlin
data class ElevationConfig(
    val smoothingWindowPoints: Int = 5,
    val minClimbDeltaMeters: Float = 3f
)
```

Логика:

```text
берём последние 5 accepted-точек с altitude
считаем среднюю высоту
получаем smoothedAltitude
```

Если точек меньше 5, можно:

- не считать climb до заполнения окна;
- либо считать по доступным точкам.

Предпочтительно: считать по доступным точкам, но только после минимум 2 точек с altitude.

---

## 5. Минимальный порог подъёма

Мелкие изменения высоты нужно игнорировать.

Правило:

```text
если smoothedAltitude выросла меньше чем на 3 м — не добавлять в total climb
если выросла на 3 м или больше — добавить разницу
```

Значение должно быть в конфиге:

```text
minClimbDeltaMeters = 3
```

---

## 6. Не учитывать отрицательные изменения

Если smoothedAltitude снизилась, total climb не уменьшается.

Пример:

```text
100 м → 95 м
```

Total climb не меняется.

---

## 7. Перезапуск накопления после спуска

Важно не терять следующий подъём.

Пример:

```text
100 → 110 → 105 → 115
```

Ожидание:

```text
+10 м
+10 м
total climb = 20 м
```

То есть после спуска новый подъём должен считаться относительно локального минимума, а не относительно предыдущего пика.

---

# Рекомендуемая модель расчёта

Нужно вести внутреннее состояние:

```kotlin
data class ElevationState(
    val smoothedAltitude: Float?,
    val lastReferenceAltitude: Float?,
    val currentMinAltitude: Float?,
    val totalClimbMeters: Float
)
```

Упрощённая логика:

```text
1. Получили accepted point с altitude.
2. Рассчитали smoothedAltitude.
3. Если это первая высота — сохранить как reference/min.
4. Если smoothedAltitude ниже currentMinAltitude — обновить currentMinAltitude.
5. Если smoothedAltitude выше currentMinAltitude:
   delta = smoothedAltitude - currentMinAltitude
6. Если delta >= minClimbDeltaMeters:
   totalClimb += delta
   currentMinAltitude = smoothedAltitude
```

Это позволяет:
- игнорировать мелкий шум;
- считать подъёмы после спусков;
- не вычитать спуски.

# UI-требования

## Экран поездки

Добавить поле:

```text
Набор высоты
0 м
```

Во время записи значение обновляется live.

Округлять до целых метров.

---

## Экран истории поездок

В карточке записи показывать:

```text
Набор высоты: 142 м
```

Если высота не считалась или данных нет:

```text
Набор высоты: —
```

---

# Хранение результата

Для каждой записи сохранять итоговый `totalClimbMeters`.


```text

```

---

# Acceptance criteria

Фича считается готовой, если:

1. Total climb считается только по accepted GPS-точкам.
2. Rejected GPS-точки не влияют на набор высоты.
3. Между сегментами подъём считается корректно.
4. Спуски не уменьшают total climb.
5. Мелкий шум altitude не завышает набор высоты.
6. После спуска следующий подъём считается корректно.
7. Значение отображается на live-экране поездки.
8. Значение сохраняется в истории поездок.
9. Altitude сохраняется в точках трека.
10. CSV содержит altitude.
11. Логика покрыта unit-тестами.
